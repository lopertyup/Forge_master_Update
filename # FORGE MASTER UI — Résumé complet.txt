### Détail P2.9 — Comparateur pré-rempli (26/04)

**Backend.** Nouveau helper `apply_change_flat_only(profile, old_eq, new_eq)` dans `backend/stats.py` qui swap UNIQUEMENT `hp_flat` / `damage_flat` / `attack_type` sans toucher aux substats du profil. Parce que les substats per-pièce ne sont pas (encore) trackés, on évite de produire un faux total en folding-back les substats du candidat.

**`compare_equipment(text, slot=None)` étendu :**
- Path 1 (legacy) : si le texte contient deux items, comportement inchangé (`apply_change` complet, substats inclus).
- Path 2 (P2.9) : si un seul item est parsé ET un `slot` est fourni, le controller :
  1. récupère le slot persisté dans `self._equipment[slot]`,
  2. construit un `old_eq` minimal `{hp_flat, damage_flat, attack_type, name, rarity, level}`,
  3. appelle `apply_change_flat_only` → swap flat-only.
- Sinon → `None` (l'UI dégrade vers un message "indique-moi le slot").

**Limitation documentée.** Sans substats per-pièce, le comparateur en path 2 ne peut PAS afficher le delta substats — le candidat affiche ses propres %, mais elles ne s'appliquent pas au total profil. Ce sera levé quand on capturera la popup long-press par pièce (hors P2).

**Tests** : 2 nouveaux dans `tests/test_compare_equipment_single_item.py` (flat-only ne touche pas aux substats / two-items continue de les swapper). 120/120 tests verts.

## 3. Attack Speed — état d'implémentation

### Ce qui est compris et implémenté

`backend/stats.py` modélise correctement le système discret :

```python
m  = 1 + attack_speed_pct / 100
sw = floor(windup   / m * 10) / 10      # Rhythmic Windup Step (implicite)
sr = floor(recovery / m * 10) / 10      # idem côté recovery
single_cycle = sw + sr + 0.2            # Primary Weapon Cycle
double_cycle = single_cycle + floor(0.25/m * 10)/10   # Double Attack Cycle
```

C'est **exactement** la mécanique des trois paliers que tu décris :

| Concept du jeu | Implémentation actuelle |
|---|---|
| Primary Weapon Cycle | `swing_time_discrete()` → `sw + sr + 0.2` |
| Rhythmic Windup Steps | Le `sw = floor(windup/m * 10)/10` reproduit exactement les paliers de la table `rhythmic_windup_steps` |
| Double Attack Cycle | `swing_time_double()` → cycle + gap steppé de 0.25 s |

### Sur le « Rhythmic Windup » spécifiquement

Tu disais que c'était flou. Voici ma lecture : c'est le **palier auquel passe la phase de windup seule**, indépendamment du cycle global. Dans les fichiers JSON, `rhythmic_windup_steps` est juste la décomposition palier-par-palier du seul windup (la table commence à des seuils plus hauts : pour le Bow, 80.6 % d'AS pour passer windup à 0.5 s, 170.8 % marqué `META` pour 0.3 s, etc.).

**C'est implicitement implémenté** par le `floor(windup/m * 10)/10` du simulateur — le windup baisse par pas de 0.1 s exactement comme dans la table, et le `META` n'est qu'un marqueur visuel d'un palier "ressenti" par l'animation.

**Ce qui n'est pas implémenté** :
- Le simulateur ne lit jamais directement les fichiers `helper/weapon atq speed/*.txt`. Le module `backend/weapon_breakpoints.py` existe (et est testé) mais aucune autre partie du backend ne l'appelle. Tout le travail est refait à partir du raw `WindupTime` / `AttackDuration` de `WeaponLibrary.json`.
- Le marqueur `META` n'est exploité nulle part — utile éventuellement pour l'UI ("tu es à 1 palier du META") mais sans impact sur la sim.

### Points de validation

Tests `test_weapon_breakpoints.py` (24 tests) et `test_projectile_simulation.py` (13 tests) verts. Ils valident la formule discrète et le routing legacy. La sim tombe en mode "linéaire legacy" si `weapon_windup` ou `weapon_recovery` manquent dans le profil — donc tant que la zone `player_weapon` n'est pas calibrée, le joueur reste sur l'ancienne formule (ATTACK_INTERVAL / speed_mult).

## 4. Audit — désalignements code ↔ vision

### ✅ P0-critique : Multiplicateurs PvP par source — CORRIGÉ (25/04)

**Avant :** `pvp_hp_total()` faisait simplement `hp_total × 5.0`, écrasant toute stratégie mount-heavy / pet-heavy.

**Après :** somme pondérée par source, conformément à `data/PvpBaseConfig.json` :
```
hp_pool_pvp = hp_equip × 1.0
            + hp_pet   × 0.5
            + hp_skill × 0.5
            + hp_mount × 2.0
```

- Sous-totaux exposés côté ennemi via `EnemyComputedStats` (4 nouveaux champs : `equip_health`, `pet_health`, `mount_health`, `skill_passive_health`).
- Sous-totaux dérivés côté joueur via le helper `compute_hp_buckets(profile, pets, mount, skills)` qui décompose `hp_base`.
- Le controller injecte les 4 buckets avant chaque `simulate()` et chaque swap-test (pets / mount / skills / equipment).
- `PVP_HP_MULTIPLIER = 5.0` conservé en **fallback legacy** uniquement (utilisé par les tests qui ne fournissent pas les 4 buckets).
- 101 / 101 tests verts.

### Formules dérivées (`backend/stats.py`)

```
speed_mult(as_pct)        = 1 + as_pct/100
crit_multi(crit_dmg_pct)  = 1.172 + crit_dmg_pct/99
swing_time_discrete       = floor(W/m * 10)/10 + floor(R/m * 10)/10 + 0.2
swing_time_double         = swing_time_discrete + floor(0.25/m * 10)/10

POST_ATTACK_FIXED         = 0.2 s   (constante interne stats.py)
DOUBLE_ATTACK_GAP         = 0.25 s  (idem)

pvp_hp_total(s)           = s.hp_equip × 1.0          (somme pondérée par source)
                          + s.hp_pet   × 0.5
                          + s.hp_skill × 0.5
                          + s.hp_mount × 2.0
                            (fallback s.hp_total × 5.0 si les 4 buckets sont absents)
compute_hp_buckets(profile, pets, mount, skills) → décompose hp_base en 4 sous-totaux
                            puis applique (1 + health_pct/100) à chacun.
pvp_regen_per_second(s)   = s.hp_total × s.health_regen / 100
                            (calculé sur le hp_total pré-PvP, tick toujours via TICK)
```

### Calculs ennemi (`backend/enemy_stat_calculator.py`)

```
item_dmg / item_hp        = Σ (EquipmentStats × 1.01^(level-1))   (1.01 = LevelScalingBase)
weapon_dmg                = item_dmg du slot Weapon
pet_dmg / pet_hp          = LevelInfo[level-1] × pet_type_multi   (Balanced 1.0/1.0,
                                                                   Tank +HP, DPS +Dmg)
mount_dmg / mount_hp      = LevelInfo[level-1]
skill_passive             = ⌊LevelStats[level-1].Damage⌋ + ⌊LevelStats[level-1].Health⌋

equip_with_melee  = is_ranged ? weapon_dmg : weapon_dmg × 1.6
                     (PlayerMeleeDamageMultiplier)

equip_dmg_pool    = (PlayerBaseDamage + equip_with_melee + (item_dmg-weapon_dmg))
                     × (1 + DamageMulti)
system_dmg_pool   = (pet_dmg + skill_passive_dmg + mount_dmg)
                     × (1 + DamageMulti)
total_damage      = (equip_dmg_pool + system_dmg_pool)
                     × (1 + (RangedDamageMulti or MeleeDamageMulti))

equip_hp_pool     = (PlayerBaseHealth + item_hp) × (1 + HealthMulti)
system_hp_pool    = (pet_hp + skill_passive_hp + mount_hp) × (1 + HealthMulti)
total_health      = equip_hp_pool + system_hp_pool

PlayerBaseDamage  = 10
PlayerBaseHealth  = 80
PlayerBaseCritDmg = 0.20
LevelScalingBase  = 1.01
```

### Combat FSM (`simulation.py:Fighter`)

```
Init :
  hp_max  = pvp_hp_total(stats)        (← 5×, à raffiner par les 4 multipliers)
  attack  = stats.attack_total
  base_swing_time   = swing_time_discrete(W, R, AS%)         si W,R connus
                      sinon legacy ATTACK_INTERVAL/speed_mult
  double_swing_time = swing_time_double(W, R, AS%)
  _double_fire_gap  = floor(0.25/speed_mult * 10)/10
  swing_timer       = -COMBAT_START_DELAY     (run-up)
  skills            = liste de SkillInstance

Tick (pas dt = 0.01 s) :
  1. Land arriving projectiles
  2. Si un combattant tombé → drain phase (projectiles seuls)
  3. Regen (snapshot par seconde, applied par tick)
  4. Skills (ordre P/O random) → tick FSM (cooldown→cast→hits)
  5. Basic attack FSM (ordre P/O random) :
       swing_timer += dt
       à expiration : roll double_chance → release 1 ou 2 hits
       block roll au tir, crit roll au tir, lifesteal stocké et appliqué à l'impact
  6. t += dt

Fin :
  Verdict : alive(P)+!alive(O) → WIN, etc.
  Timeout : compare hp_pct() avec ε
```

**Objectif.** Aujourd'hui le profil joueur (`profile.txt`) ne stocke que des

2. empêche le simulateur de répliquer fidèlement la formule TS du jeu

   tracking du progrès, alertes "tu as oublié d'upgrader telle pièce").

**Approche.** L'identification d'icônes + OCR du level + lookup
`ItemBalancingLibrary` est DÉJÀ implémentée et testée côté ennemi

- Factoriser ce pipeline pour qu'il prenne aussi un screenshot côté
  joueur (panel "Equipement" du jeu — 8 slots, layout différent du

- Persister les 8 IdentifiedItem du joueur dans un nouveau fichier

**Recognition strategy — icône d'abord, OCR du nom en backup.**

| Source           | Confiance | Coût            | Décision |

| OCR du Lv.NN sous l'icône | indispensable pour le scaling | déjà fait par `_extract_level` | **toujours** |

Le `[Rarity]` et le `[Age]` sont aussi extraits par parsing du nom — utiles
pour valider la rarity inférée par couleur.
