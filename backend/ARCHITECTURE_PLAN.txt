────────────────────────────────────────────────────────────────────────────────
0. VISION PRODUIT (rappel)
────────────────────────────────────────────────────────────────────────────────

L'app est un assistant PvP pour un jeu mobile. Le jeu s'affiche à droite de
l'écran, l'interface Forge Master à gauche. L'objectif central est de simuler
des combats avec un maximum de précision pour aider le joueur à prendre des
décisions (quel équipement garder, quel pet monter, etc.).

Sections de l'app :

  1. Profile / Dashboard — toutes les infos du joueur (stats, pets, équipement,
     monture, skills avec leurs substats). Source de vérité du profil.

  2. Simulator — scanne un ennemi, identifie ses stats depuis les JSON du jeu,
     simule 1 000 combats PvP et affiche un pourcentage de victoire.

  3. Equipment — sauvegarde l'équipement joueur. Quand il change une pièce,
     simule ancien vs nouveau et donne la recommandation.

  4. Pets / Mount / Skills — même logique qu'Equipment : swap simulation
     comparant la situation actuelle vs un candidat.

  INVARIANT GLOBAL : toute simulation utilise les stats PvP officielles
  (multiplicateurs PvP sur les HP, formule de cycle d'attaque discrète,
  délai de départ de combat, délai initial de skill). Aucune approximation.

2. PROBLÈMES IDENTIFIÉS
────────────────────────────────────────────────────────────────────────────────

  P1 — Nommage trompeur "enemy_*"
       enemy_icon_identifier, enemy_icon_offsets, enemy_libraries, etc. sont
       TOUS utilisés par le pipeline joueur aussi. Le préfixe "enemy" cache
       le fait que c'est du code générique.

  P2 — Couplage croisé privé→privé
       player_equipment_scanner et player_weapon_scanner importent _item_key()
       et _stat_type() depuis enemy_stat_calculator — deux fonctions privées
       (underscore) qui traversent une frontière de module. Fragile.

  P3 — "Dieu module" enemy_pipeline
       enemy_pipeline.py importe depuis 7 modules différents et orchestre
       tout. Trop de couplage en un seul endroit.

  P4 — Flat plat de 25 fichiers
       Impossible de savoir d'un coup d'œil ce qui est "calcul pur" vs "I/O
       image" vs "simulation". Navigation difficile pour un nouveau contributeur.

  P5 — forge_master.py shim inutile
       Ce fichier ne sert qu'à éviter de mettre à jour les imports. Il masque
       la vraie structure et doit être supprimé.

  P6 — attack_speed_calculator charge WeaponLibrary lui-même
       Il a son propre cache _wl_cache au lieu d'utiliser le chargeur central
       (enemy_libraries / data/libraries.py). Deux chemins de lecture du même
       fichier JSON.

  P7 — weapon_breakpoints charge ses propres fichiers JSON
       Même problème : chargeur isolé sans passer par le module data central.

5. CAS PARTICULIER : parser.py
────────────────────────────────────────────────────────────────────────────────

  parser.py contient deux choses distinctes :

  a) Des parsers de texte OCR (parse_profile_text, parse_companion,
     parse_equipment, parse_skill_meta…) — c'est de la logique scanner.
     → Déplacer dans scanner/text_parser.py

  b) Des utilitaires bas niveau (parse_flat, extract, extract_flat) —
     utilisés partout, y compris dans enemy_ocr_parser.
     → Déplacer dans un module utilitaire scanner/utils.py
     (ou rester dans scanner/text_parser.py et être importés depuis là).

  L'import parse_flat dans enemy_ocr_parser.py devient :
      from .text_parser import parse_flat   # (ou from ..scanner.utils)

  Le module parser.py est ensuite SUPPRIMÉ. Tous les appelants
  (persistence, controller, UI) mettent à jour leurs imports.

7. POINTS DE VIGILANCE TECHNIQUE
────────────────────────────────────────────────────────────────────────────────

  V1 — PvP stats : invariant critique
       Toute simulation DOIT utiliser pvp_hp_total() (calculator/stats.py) et
       les multiplicateurs PvP de constants.py. Aucun code ne doit utiliser
       hp_total brut en combat. À vérifier dans engine.py après déplacement.

  V2 — Chargeur JSON unique
       Après les phases 1-3, il ne doit exister QU'UN chargeur JSON actif :
       data/libraries.load_libs(). Les caches isolés dans attack_speed_calculator
       et weapon_breakpoints doivent être supprimés ou branchés sur load_libs().

  V3 — Imports relatifs
       Après réorganisation, tous les imports DOIVENT être relatifs (from . ou
       from ..). Aucun "from backend.xxx import" dans le code interne — cela
       casse dès qu'on renomme un dossier.

  V4 — debug_scan chemin root
       debug_scan.py calcule _ROOT_DIR en remontant depuis backend/. Après
       déplacement dans scanner/, il faudra remonter d'un niveau supplémentaire
       (parent.parent au lieu de parent).

  V5 — icon_recognition.py chemin DATA_DIR
       Même problème de chemin relatif. Vérifier après déplacement dans scanner/.

  V6 — forge_master.py : identifier tous les consommateurs avant suppression
       Chercher "from .forge_master import" ou "from backend.forge_master import"
       dans toute la codebase (UI comprise) avant de supprimer.

  V7 — Skill cooldown négatif
       Dans constants.py et optimizer.py, skill_cooldown est un stat dont les
       valeurs négatives signifient "réduction". Le simulateur doit traiter ça
       correctement. Vérifier que engine.py lit le bon champ après déplacement.
