================================================================================
  FORGE MASTER - PLAN DE REFONTE DU SYSTEME DE SCAN ET DE PERSISTANCE
  Version corrigee apres revue humaine
================================================================================

Ce document est une specification de travail. Il doit guider une refonte
complete, mais chaque phase doit rester compilable et testable.

Vision cible validee :

  1. `data/` a la racine du projet est la source de verite pour les donnees
     runtime partagees :
       - `data/colors.json`
       - `data/libraries.py`
       - `data/library_ops.py`
       - `data/icons/...`
       - futures librairies de nomenclature canonique.

  2. Les scans joueur sont OCR purs :
       - equipment_popup
       - player_equipment
       - pet
       - mount
       - skill

     Ces jobs ne doivent plus faire d'identification visuelle par icone,
     couleur ou reference image. Ils lisent le texte OCR, normalisent, parsers,
     puis construisent un dict canonique.

  3. Seul le scan adversaire garde la strategie visuelle :
       - scan.core
       - scan.colors
       - scan.refs
       - data/icons

     Raison : sur l'adversaire, les infos OCR disponibles ne suffisent pas.
     Les icones, couleurs et levels restent necessaires.

  4. La persistance joueur est refondue completement. L'ancien fonctionnement
     profile.txt + equipment.txt + pets.txt + mount.txt + skills.txt devient
     caduc apres migration. Toute sauvegarde passe par un nouveau store profile
     centralise dans `backend/persistence/profile_store/`.

  5. Les nouveaux noms canoniques deviennent la norme :
       - Equipment slots : Helmet, Body, Gloves, Necklace, Ring, Weapon, Shoe, Belt
       - Skill slots     : Skill_1, Skill_2, Skill_3
       - Pet slots       : Pet_1, Pet_2, Pet_3
       - Mount slot      : Mount

     Ces noms doivent aussi s'aligner sur `data/icons/equipment/<Age>/<Slot>/`.

================================================================================
  0. REGLES GLOBALES
================================================================================

R0. Avant chaque etape, lire les fichiers cites dans l'etape.

R1. Une etape = un commit logique. Ne pas melanger persistance, OCR, UI et
    deplacement de modules dans le meme commit.

R2. Apres chaque etape, executer au minimum :

      python -m compileall backend scan ui data game_controller.py main.py
      python -c "import game_controller"

    Si pytest est disponible :

      python -m pytest

R3. Ne jamais supprimer un fichier appele par le code avant d'avoir migre tous
    ses callers ou ajoute un shim temporaire.

R4. Les anciens fichiers de persistance peuvent devenir caducs, mais leur
    migration doit etre explicite, testee et sauvegarder l'existant en `.bak`.

R5. Ne jamais modifier les JSON de librairie metier par accident. Les fichiers
    `data/*.json` sont des donnees runtime, pas des sorties de tests.

R6. Les jobs scan ne font jamais de Tk, jamais de popup UI, jamais de save
    direct. Ils retournent un `ScanResult`.

R7. Toute sauvegarde joueur passe par le controller, puis par le profile store.

================================================================================
  2. INVARIANTS NON NEGOCIABLES
================================================================================

V1. Skill Cooldown est un float signe. Jamais `abs()`, jamais cast positif.

V2. Pet et Mount lisent toujours les deux stats principales :
    `hp_flat` ET `damage_flat`.

V3. Les substats sont variables : 0..N entrees. Le parser ne doit jamais
    supposer un nombre fixe de lignes.

V4. Les scans de comparaison n'ecrivent rien. Ils simulent et renvoient le
    resultat. La sauvegarde attend une confirmation utilisateur.

V5. Jobs joueur OCR purs :

      scan/jobs/equipment_popup.py
      scan/jobs/player_equipment.py
      scan/jobs/pet.py
      scan/jobs/mount.py
      scan/jobs/skill.py

    Interdits dans ces fichiers :

      from scan.core ...
      from scan.colors ...
      from scan.refs ...
      from ._flat import run_flat_scan
      from ._panel import identify_panel
      from ..core ...
      from ..colors ...
      from ..refs ...

    Exception temporaire autorisee pendant transition :
      importer une constante canonique depuis `data.canonical`.

V6. Seul `scan/jobs/opponent.py` peut utiliser core/colors/refs pour identifier
    visuellement icones, ages et rarities.

V7. OpenCV reste optionnel. `scan.core` doit pouvoir etre importe sans cv2.

V8. Les noms canoniques de slots sont ceux de `data.canonical`, pas les anciens
    `EQUIP_*`, `PET1`, `S1`.

V9. Toute ecriture profile recalcule les totaux derives et substats totals.

V10. Les skills ne contribuent pas a `substats_total`, mais leurs passifs HP et
     Damage contribuent aux totaux de combat si le simulateur les utilise deja.

V11. `game_controller.py` est la frontiere : UI -> controller -> scan/backend.
     Les vues UI ne sauvegardent pas directement.

V12. Les anciens modules `backend.scanner.*` doivent rester shimes tant qu'un
     import direct existe.

================================================================================
  4. PHASE B - PERSISTANCE PROFILE CENTRALISEE
================================================================================

But :
  Remplacer l'ancien systeme eclate par un seul store profile centralise.

Ancien systeme :
  - backend/profile.txt
  - backend/equipment.txt
  - backend/pets.txt
  - backend/mount.txt
  - backend/skills.txt

Nouveau systeme :
  - backend/persistence/profile_store/profile.txt
  - backend/persistence/profile_store/profile_meta.json optionnel

Format Python canonique :

  profile = {
      "equipment": {
          "Helmet": {
              "__name__": str,

              "__rarity__": str,

              "hp_flat": float,

              "weapon_attack_range": float,

================================================================================
  5. PHASE C - JOBS SCAN JOUEUR OCR PURS
================================================================================

But :
  Supprimer la dependance des scans joueur a l'identification visuelle.
  L'OCR doit fournir toutes les informations necessaires.

Parser cible :

  scan/ocr/parsers/equipment.py
    parse_equipment_popup_text(text: str, *, slot: str) -> dict

  scan/ocr/parsers/companion.py
    parse_companion_text(text: str) -> dict

  scan/ocr/parsers/skill.py
    parse_skill_text(text: str) -> dict

  Chaque parser retourne un dict canonique directement compatible avec le
  profile store.

Regles communes :
  - `ocr_image(capture)` lit le texte.
  - `fix_ocr(text, context=...)` normalise.
  - Parser dedie extrait name, level, rarity/age, hp_flat, damage_flat,
    substats et type si skill.
  - Les substats sont mappees via `data.canonical.SUBSTAT_ALIASES`.
  - Le job retourne `ScanResult(status=..., matches=[], debug={"slot_dict": ...})`.
  - Pas de `scan.core`, pas de `scan.refs`, pas de `scan.colors`.

equipment_popup.py :
  - `force_slot` obligatoire.
  - OCR lit le popup detail.
  - La balise `[Age]` donne `__age__`.
  - Le nom OCR donne `__name__`.
  - Le level OCR donne `__level__`.
  - HP/Damage/substats viennent du texte.
  - Si `slot == "Weapon"`, appeler `enrich_weapon_slot`.
  - Ne pas matcher l'icone.
  - Ne pas lire la couleur.

player_equipment.py :
  - Decouper le panel en 8 zones selon `scan/offsets/player.py`.
  - Pour chaque slot canonique, OCR de la zone correspondante.
  - Parser comme equipment.
  - Si un slot est incomplet, garder le slot avec `status="low_confidence"`
    et debug detaillant les champs manquants.
  - Ne pas appeler `_panel.identify_panel`.

pet.py et mount.py :
  - OCR popup.
  - Lire rarity, name, level, HP, Damage, substats.
  - Construire un slot dict canonique.
  - Ne pas appeler `_flat.run_flat_scan`.

skill.py :
  - OCR popup.
  - Lire rarity, name, level.
  - Lire passifs :
      "+X Base Damage" -> damage_flat
      "+X Base Health" -> hp_flat
  - Determiner `type` :
      active damage detecte -> "damage"
      sinon -> "buff"
  - `substats` doit etre `{}`.
  - Ne pas appeler `_flat.run_flat_scan`.

_flat.py et _panel.py :
  - Garder temporairement si utile pour adversaire/debug.
  - Ajouter un commentaire :
      "Interdit aux jobs joueur. Usage adversaire ou legacy/debug uniquement."
  - A terme, si aucun caller hors adversaire ne les utilise, les isoler ou les
    supprimer.

Verification stricte :

  rg -n "from \\.\\.core|from \\.\\.colors|from \\.\\.refs|from \\.core|from \\.colors|from \\.refs|_flat|_panel" scan/jobs/equipment_popup.py scan/jobs/player_equipment.py scan/jobs/pet.py scan/jobs/mount.py scan/jobs/skill.py

  Doit etre vide, sauf commentaires temporaires explicitement acceptes.


================================================================================
  6. PHASE D - CONTROLLER ET UI
================================================================================

But :
  Faire passer toutes les operations joueur par le nouveau profile store.

game_controller.py cible :
  - `self._profile_store` ou `self._profile` contient le dict schema v2 complet.
  - `get_equipment()`, `get_pets()`, `get_mount()`, `get_skill_slots()`
    retournent des vues derivees du profile complet.
  - `set_equipment_slot(slot, data)`
  - `set_pet_slot(slot, data)`
  - `set_mount(data)`
  - `set_skill_slot(slot, data)`
  - Chaque setter :
      1. merge dans profile
      2. recalcule substats_total
      3. save_profile(profile)
      4. notifie les listeners

Events a conserver/ajouter :
  - equipment_changed
  - pets_changed
  - mount_changed
  - skills_changed
  - profile_changed si utile

Cas d'usage :

  Cas 1 - comparaison double equipment :
    1. scanner l'item equipe par OCR pur
    2. scanner l'item candidat par OCR pur
    3. simuler le remplacement
    4. callback vers UI
    5. aucune sauvegarde avant confirmation

  Cas 2 - scan simple d'un slot equipment :
    1. equipment_popup.scan(..., force_slot=slot)
    2. set_equipment_slot(slot, slot_dict)
    3. sauvegarde immediate car action explicite sur ce slot

  Cas 3 - pet/mount/skill :
    1. scan OCR pur
    2. simuler sur les slots possibles
    3. callback vers UI
    4. sauvegarde seulement quand l'utilisateur choisit/applique

UI :
  - Aucune vue ne doit appeler directement save_*.
  - Les vues appellent le controller.
  - Aucune popup ne demande Age, Rarity ou Slot si l'OCR les donne.
  - Si OCR incomplet, afficher low_confidence avec champs manquants.

Verification :
  rg -n "save_equipment|save_pets|save_mount|save_skills|save_profile" ui game_controller.py

  Les appels `save_profile` doivent etre limites au nouveau chemin controller
  centralise ou au profile store.
